# 正式学术风格写作指南

本指南用于规范需求文档的语言风格，确保文档的专业性和一致性。

## 一、语言风格原则

### 1.1 正式性

**避免：**
- 口语化表达（"搞定"、"弄一下"、"整一整"）
- 缩写形式（"can't" → "cannot"、"won't" → "will not"）
- 感叹号和表情符号
- 主观评价词（"非常好"、"超级"、"极其"）

**使用：**
- 规范的专业术语
- 完整的句子结构
- 客观的描述性语言

### 1.2 精确性

**避免：**
- 模糊表述（"可能"、"也许"、"大概"、"一些"）
- 不确定的数量词（"很多"、"少量"、"频繁"）

**使用：**
- 具体的数据和指标
- 明确的时间节点
- 可量化的描述

### 1.3 简洁性

**避免：**
- 冗余的修饰词
- 重复的表述
- 过长的句子（超过30字）

**使用：**
- 简洁的陈述
- 结构化的列表
- 表格呈现对比信息

---

## 二、常用表达规范

### 2.1 动词使用

| 非正式 | 正式 |
|--------|------|
| 搞定 | 完成、实现 |
| 弄 | 处理、操作 |
| 看一下 | 审查、检查 |
| 改一下 | 修改、调整 |
| 写一下 | 编写、撰写 |
| 跑一下 | 执行、运行 |

### 2.2 术语规范

| 非正式 | 正式 |
|--------|------|
| Bug | 缺陷、故障 |
| 需求 | 需求规格 |
| 功能 | 功能特性 |
| 接口 | 应用程序接口（API） |
| 数据 | 数据结构、数据模型 |

### 2.3 句式规范

**问题陈述：**
- 非正式："这个功能有问题"
- 正式："该功能存在以下缺陷：..."

**目标描述：**
- 非正式："我们要做一个新功能"
- 正式："本需求的目标是实现..."

**方案说明：**
- 非正式："可以这样弄"
- 正式："建议采用以下方案：..."

## 三、文档结构规范

### 3.1 标题层级

```
# 一级标题（文档标题）
## 二级标题（章节）
### 三级标题（小节）
#### 四级标题（细分）
```

### 3.2 列表格式

**无序列表：** 用于并列关系
- 项目一
- 项目二

**有序列表：** 用于顺序关系
1. 第一步
2. 第二步

**任务列表：** 用于验收标准
- [ ] 待完成项
- [x] 已完成项

### 3.3 表格格式

| 列标题1 | 列标题2 | 列标题3 |
|---------|---------|---------|
| 内容1   | 内容2   | 内容3   |

## 四、示例对比

### 4.1 需求背景

**非正式：**
```
现在用户登录很慢，经常报错，用户体验不好，需要优化一下。
```

**正式：**
```
### 1.1 现状分析

当前用户登录模块存在以下问题：

1. 登录响应时间过长，平均耗时 3.5 秒，超出系统设计目标（1 秒）
2. 高并发场景下错误率上升至 5%，主要表现为连接超时
3. 用户反馈统计显示，登录相关问题占总投诉量的 23%

### 1.2 问题陈述

上述问题严重影响用户首次使用体验，导致新用户流失率增加约 15%。
```

### 4.2 功能描述

**非正式：**
```
做一个搜索功能，可以搜各种东西，还要支持模糊搜索。
```

**正式：**
```
### 3.1 功能概述

系统需提供全局搜索功能，支持以下搜索能力：

| 功能编号 | 功能名称 | 功能描述 | 优先级 |
|----------|----------|----------|--------|
| F001     | 关键词搜索 | 支持按关键词检索所有业务数据 | P0 |
| F002     | 模糊匹配   | 支持拼音、首字母、同义词匹配 | P1 |
| F003     | 高级筛选   | 支持按时间、类型、状态等维度筛选 | P1 |
```
