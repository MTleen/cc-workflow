# 术语表

本文档定义 CC-Workflow 中的核心术语。

---

## 核心概念

### Skill

**定义**：流程规范层，定义「做什么」和「输出什么」。

**位置**：`.claude/skills/`

**特点**：
- 定义阶段目标、输入输出、模板、检查清单
- 约束执行流程、评审条件
- 不定义角色身份（由 Agent 定义）

**示例**：
- `ideal-requirement`：P1 需求编写
- `ideal-dev-solution`：P3 技术方案
- `ideal-wiki`：P15 维基更新

**相关概念**：[Agent](#agent)

---

### Agent

**定义**：角色能力层，定义「如何思考」。

**位置**：`.claude/agents/`

**特点**：
- 定义角色身份、专业视角、思维方式
- 可复用、可组合、独立于流程
- 不定义输出模板（由 Skill 定义）

**示例**：
| Agent | 角色 | 思维方式 |
|-------|------|----------|
| pm | 产品经理 | 用户价值导向、MVP 思维 |
| architect | 架构师 | 系统思维、权衡取舍 |
| dev | 开发工程师 | TDD、代码质量 |
| qa | 测试工程师 | 边界思维、风险意识 |

**相关概念**：[Skill](#skill)

---

### 故事文件（Story File）

**定义**：P5 阶段生成的原子化任务单元，用于 P9 开发执行时的上下文隔离。

**位置**：`docs/迭代/{需求名}/stories/`

**结构**：
```
stories/
├── index.md           # 故事索引
├── 001-xxx.md         # 故事 1
├── 002-xxx.md         # 故事 2
└── ...
```

**作用**：
- P9 开发时只加载当前故事文件
- 避免加载无关的上下文信息
- 降低 Token 消耗 70-90%

---

### 流程状态文件

**定义**：控制流程阶段流转的 YAML 文件。

**位置**：`docs/迭代/{需求名}/流程状态.md`

**格式**：
```yaml
---
current_phase: P6
requirement_review: approved
plan_review: pending
stories_dir: docs/迭代/用户登录/stories/
---
```

**核心字段**：
| 字段 | 说明 |
|------|------|
| `current_phase` | 当前阶段（P1-P16） |
| `xxx_review` | 评审状态（pending/approved/rejected） |
| `stories_dir` | 故事文件目录路径 |

## 流程阶段

### P1-P16

**定义**：CC-Workflow 将软件开发流程划分为 16 个阶段。

**分组**：
| 分组 | 阶段 | 说明 |
|------|------|------|
| 规划阶段 | P1-P4 | 需求收集和技术方案设计 |
| 准备阶段 | P5-P8 | 计划生成和测试用例准备 |
| 执行阶段 | P9-P12 | 代码开发和测试验证 |
| 收尾阶段 | P13-P16 | 上线决策和文档更新 |

---

### 评审节点

**定义**：需要人工决策的阶段节点。

**列表**：
| 阶段 | 评审内容 |
|------|----------|
| P2 | 需求文档 |
| P4 | 技术方案 |
| P6 | 编码计划 |
| P8 | 测试用例 |
| P10 | 代码 MR |
| P12 | 测试报告 |
| P13 | 上线决策 |
| P16 | Wiki 文档 |

---

## 文件类型

### P1-需求文档.md

**定义**：P1 阶段输出的需求文档。

**内容**：
- 需求背景
- 问题陈述
- 目标概述
- 功能需求
- 非功能需求
- 验收标准

---

### P3-技术方案.md

**定义**：P3 阶段输出的技术方案文档。

**内容**：
- 系统架构设计
- 技术选型说明
- 模块划分
- 风险评估
- 实现策略

---

### P5-编码计划.md

**定义**：P5 阶段输出的编码计划文档。

**内容**：
- 任务拆分
- 依赖关系
- 排期计划
- 技术要点

---

### P7-测试用例.md

**定义**：P7 阶段输出的测试用例文档。

**内容**：
- 功能测试用例
- 边界测试用例
- 异常测试用例
- 测试数据

---

### P11-测试报告.md

**定义**：P11 阶段输出的测试报告文档。

**内容**：
- 测试执行结果
- 通过率统计
- 缺陷记录
- 遗留问题

---

## 协作模式

### 人机协作

**定义**：团队成员与 Claude Code 的协作模式。

**分工**：
| 角色 | 职责 |
|------|------|
| 团队成员 | 提需求、做评审、反馈意见 |
| Claude Code | 需求编写、技术方案、开发执行、测试执行、维基更新 |

---

### 苏格拉底式对话

**定义**：P1 需求收集阶段采用的对话方式。

**特点**：
- 通过提问引导用户思考
- 逐步深入挖掘需求细节
- 避免假设，确保理解准确

**问题类型**：
- 问题背景
- 目标用户
- 功能细节
- 验收标准

---

### TDD（测试驱动开发）

**定义**：P9 开发执行阶段采用的开发方式。

**流程**：
1. 先写测试用例（测试失败）
2. 编写最小实现代码（测试通过）
3. 重构代码（保持测试通过）

---

## 其他术语

### MR（Merge Request）

**定义**：GitLab 的合并请求，用于代码评审和合并。

**相关阶段**：P9 创建，P10 评审

---

### CI/CD

**定义**：持续集成/持续部署，P14 阶段的自动化部署流程。

---

### Obsidian

**定义**：Markdown 笔记软件，用于查看和编辑 CC-Workflow 的文档。

**插件**：Obsidian Git 插件可自动同步文档到 Git 仓库。
